## Wave 8 closure

1. **Reconciled**: all 152 states are accounted for above; none omitted. 0 rows are `POLICY_HIDDEN` or `OBSOLETE_BY_CONTRACT` in this pass -- every gap found was either a real frontend build gap (now closed, 65 -> 5 `MISSING` across Waves 0-7) or a genuine backend contract gap (3 rows, all in `backend-contract-blockers.md` with exact missing operation, request/response shape, and recommended owner).
2. **Remaining PARTIAL/MISSING rows** (10 + 5 = 15) all carry a specific reason in their notes column, not a placeholder. The largest cluster is `G5-CHK-13/14/15`: the Zarinpal callback intentionally returns one merged `cancelled_or_failed` state (`GATE52C_RESIDUAL` #1, an accepted design residual, not a bug); the frontend correctly mirrors that merge rather than inventing a distinction the backend doesn't make. The rest (`G5-SHOP-05/13/14` product compare/alternatives/search, `G5-AUTH-02` a sub-second loading transition, `G5-BRIDGE-03/13/14` edge-case pet-id fallback / background-refetch indicator / sheet-vs-page presentation, `G5-ACC-03/16` backend-blocked address edit and privacy-status read-back) are each either explicitly deferred in the design source (`disposition: deferred/local`) or blocked on the backend gaps already recorded.
3. **No simulated T8-T18 behavior remains in production code** -- verified by grepping `src/app`, `src/features`, `src/components`, `src/lib` for TODO/FIXME/simulate/mock/fake outside test and fixture files: no matches. The one fixture-backed code path (`src/lib/api/dev-fixtures.server.ts`) is gated behind `NODE_ENV !== "production" && GATE_FIXTURE_MODE === "1"` and is never reachable in a production build.
4. **No dead scaffold pages found.** `/inventory/open` is kept deliberately as a legacy-URL redirect (now to `/inventory`, the real list page) rather than removed, since it may be a bookmarked/shared link target; it contains no other logic.
5. **Loading/empty/error/fatal states, responsive layout, keyboard nav, and screen-reader semantics** are provided by the shared primitives (`src/components/primitives.tsx`) used consistently across all 152 states' implementations: `Skeleton` for loading, `EmptyState`/`ErrorState` for empty and recoverable-error, the root `error.tsx`/`not-found.tsx` for fatal/unmatched routes, logical CSS properties and the 320/360/768px breakpoints from Wave 0 for responsive layout, and `Sheet`/`Dialog`'s focus-trap-with-restore plus visible `:focus-visible` outlines and `prefers-reduced-motion` handling for keyboard/screen-reader/motion. This was not re-audited route-by-route in Wave 8 beyond the checks already run per wave (typecheck, lint, unit/component tests, production build) -- an explicit axe-core pass and manual screen-reader walkthrough were not performed and would be the natural next step before a release gate.

## Post-Wave-8 correctness pass (2026-07-18)

A follow-up audit against the live backend/OpenAPI found four real defects behind states that were already marked `IMPLEMENTED`; all four are now fixed, tested (real Postgres integration tests plus component tests), and reflected in the row notes above:

1. **G5-JOURNEY-12/18 (journey auto-completion)**: the backend can complete a journey inline on the check-in that satisfies `completion_requires`, returning `completed`/`diary_entry_id`/`garden_reward_id` on `JourneyCheckInResponse` -- the frontend discarded that response, so the diary/Garden-reward transition never surfaced. Fixed by deriving the completion card from `JourneyDetailResponse` (now carrying `diary_entry_id`/`garden_reward_id` when `status=completed`), and by making the backend's replay/concurrent-conflict paths on `POST .../check-ins` report the journey's real completion state instead of a stale default.
2. **Pet-health consent (Wave 6 addendum, out-of-scope-by-design but present in shipped code)**: consent was granted silently as a side effect of the upload click, using a hardcoded `policy_version` never actually confirmed against the backend. Fixed: `pet_health_consent_policy_version` is now a real backend-governed policy field on `GET /system/policies`; a new `GET /pet-life/pets/{pet_id}/consents` lets the frontend read existing consent state; and `pet-assets.tsx` now shows an explicit Persian consent screen requiring an affirmative action before any upload UI is reachable, with stale-policy-version handling and a working withdrawal action.
3. **Untyped health API contracts**: every pet-health/knowledge/pet-assets response that was `dict[str, object]` / `additionalProperties: true` in the OpenAPI (~25 operations) is now an explicit, named Pydantic schema, including a discriminated union for weight-trend and pet-knowledge. All 12 `as unknown as Promise<...>` casts in `backend.ts` are gone, and the hand-maintained `src/lib/pet-health-types.ts` was deleted in favor of generated types.
4. **Price-observation transaction proof (backend, not a state row)**: `insert_observation`'s conflict handling already never rolled back; added integration coverage proving an outer transaction survives a duplicate-conflict alongside other writes, that concurrent replay produces exactly one row, and that the collection worker's `prices_inserted` counts only new observations while surviving a per-product parse failure. That last test surfaced two real, pre-existing schema defects (missing `server_default` on `created_at`/`updated_at` across all 8 `price_intelligence_*` tables, and a `valid_match_method` CHECK constraint that rejected the legitimate `unmatched` value) -- both repaired in migration `20260717_0025`.

Sections 4-13 of the originating full-platform mission brief (shop discovery, checkout/payment recovery states, notifications/wallet/privacy persistence, multi-pet order-line planning, Garden asset-rule completion, real-backend Playwright coverage for all 11 journeys, and a full accessibility audit) were not attempted in this pass; they remain open scope, not silently dropped.
